40// Generally floating point colour representations are considered to be in linear colour-space. Linear is where you 
41// want to do all the work. sRGB (gamma corrected) is probably the space of most textures, especially diffuse textures 
42// as they were authored on monitors that had a non-linear (gamma) response. Use this enum to indicate 
43// the colour-space of pixel data you have... if you know it. Unfortunatly you can't in general determine the space 
44// from, say, the pixel format -- a non sRGB format may contain sRGB data (but an sRGB format should be assumed to 
45// actually contain sRGB data). In the below enumerants: 
46// l = linear 
47// g = gamma 
48// q = square 
49// s = standard 
50// See tColourProfile for an enum that describes which spaces are used for which channels, and whether they are LDR or 
51// HDR, in image files. 
52enum class tColourSpace 
53{ 
54 Unspecified, 
55 
56 // Linear. 
57 // Colours (and alpha) represented in this space can be added and multiplied with each other. This is your basic 
58 // RGB cube. Alphas are also considered linear. 
59 lRGB, lRGBA = lRGB, 
60 
61 // Gamma. 
62 // Colours can be multiplied with each other, but not added. This is a common approximation of sRGB-space in 
63 // which a simple pow function is used with a nominal gamma value of 2.2. If the data contains alpha, the alpha 
64 // is considered to be linear. 
65 gRGB, gRGBlA = gRGB, 
66 
67 // Square. 
68 // This is a lame approximation of gamma-space in which gamma is taken to be 2.0. This allows fast conversion 
69 // between linear and gamma-square because a square and square-root function are all that's needed. 
70 qRGB, qRGBlA = qRGB, 
71 
72 // Standard. 
73 // Standard RGB. This is the real-deal and uses the full sRGB spec (https://en.wikipedia.org/wiki/SRGB) 
74 // Neither mult or add. Most common space of source art. 
75 sRGB, sRGBlA = sRGB, 
76 
77 // I don't think we're going to need it, but we could add sRGBA if needed, where A is also in standard space. 
78  
79 // Hue, Saturation, and Value. 
80 HSV, 
81 
82 NumSpaces, 
83 
84 // Synonyms. 
85 Invalid = Unspecified, 
86 Linear = lRGB, 
87 Gamma = gRGB, 
88 Square = qRGB, 
89 Standard = sRGB, 
90 Auto = NumSpaces 
91};

94// The colour profile specifies the possible variants of how pixel data may be stored. In Tacent the colour profile is 
95// stored separately to the pixel-format. It specifies how the RGBA components should be intyerpreted, not how they're 
96// encoded. This includes both what space each component is in as well as range information: LDR (low-dynamic-range) or 
97// HDR (high-dynamic-range). 
98// 
99// For example, many RGBA images have the RGB components in sRGB-space, but the A component in linear. We use the term 
100// LDR rather than the UNORM term you see in many pixel-formats because UNORM also implies that the data is stored as 
101// an integer: UNORM -> unsigned normalized integer where all bits 0 represents 0.0 and all bits 1 represents 1.0. 
102// There's nothing stopping a pure float/half-float encoding being limited to the range 0.0 to 1.0. 
103// The profile enumerants use the following mnemonics: 
104// 
105// LDR : Low dynamic range. Values restricted to [0.0, 1.0]. 
106// HDR : High dynamic range. Values are in [0.0, infinity]. 
107// s : Standard colour space. AKA sRGB. 
108// g : Similar to Standard colour space but uses a simple pow function with gamma. 
109// l : That's an 'L'. Linear colour space. In practice HDR only ever uses linear. 
110// RGBA : These letters are used for the various compenents that may be present. 
111// 
112// For cases where a pixel format does not encode all 4 components, the first profile that fits should be used. 
113// For example a .rgb (Radiance) file does not store alpha, but does store RGB as linear HDR. The profile would be 
114// HDRlRGB_LDRlA rather than HDRlRGBA. Note also that not all tColourSpaces are represented in the profiles. This is 
115// because the profiles are used for what is commonly found in image pixel data, which does not include all the spaces 
116// mentioned in the tColourSpace enum. 
117enum class tColourProfile 
118{ 
119 Unspecified = -1, 
120 LDRsRGB_LDRlA, // Equiv ASTC profile: ASTCENC_PRF_LDR_SRGB. The LDR sRGB colour profile. 
121 LDRgRGB_LDRlA, // No ASTC profile equivalent. The LDR gRGB colour profile. 
122 LDRlRGBA, // Equiv ASTC profile: ASTCENC_PRF_LDR. The LDR linear colour profile. 
123 HDRlRGB_LDRlA, // Equiv ASTC profile: ASTCENC_PRF_HDR_RGB_LDR_A. The HDR RGB with LDR alpha colour profile. 
124 HDRlRGBA, // Equiv ASTC profile: ASTCENC_PRF_HDR. The HDR RGBA colour profile. 
125 Auto, 
126 NumProfiles, // Includes Auto but not Unspecified. 
127 
128 // Shorter synonyms. 
129 Invalid = Unspecified, 
130 None = Unspecified, 
131 sRGB = LDRsRGB_LDRlA, 
132 gRGB = LDRgRGB_LDRlA, // Currently not found in files, but may be converted to. 
133 lRGB = LDRlRGBA, 
134 HDRa = HDRlRGB_LDRlA, 
135 HDRA = HDRlRGBA, 
136};

167// ChannelType is additional satellite information that is not entirely specified by the pixel format so it belongs as 
168// satellite information here. In particular the part that isn't specified is whether the component data of each colour 
169// channel should be normalized or not afterwards. It gets a little tricky here because Vulkan, OpenGL, and DirectX 
170// have all decided on variant pixel-format names with channel-type information like UNORM, SNORM, UINT, SINT, and 
171// FLOAT. This naming _includes_ both information about how the data is encoded (integer or float, signed or unsigned) 
172// as well as whether to normalize after decoding or not. We have a choice here, either ONLY make this satellite info 
173// contain whether to normalize of not afterwards, or have a litte redundant information in order to keep the naming 
174// as close as possible to UNORM, UINT, etc. I have decided on the latter. 
175// 
176// The reason it is not part of the pixel format is it is quite common for the data to be encoded as, say, an unsigned 
177// integer, but 'converted' to a float when it is passed off the video memory by the graphics API so it is available as 
178// a float in the fragment/pixel shader. In short the ChannelType indicates intent for what should happen to the 
179// value AFTER decoding. For example, UNORM means the data is stored (or decoded for compressed formats) as an unsigned 
180// integer (which is already known by looking at the pixel-format) -- it is then converted to a normalized value in 
181// [0.0, 1.0]. SNORM means it's stored as a signed integer and then normalized to the [0.0, 1.0] range. The actual 
182// number of bits used is NOT specified here -- that is also specified by the pixel-format itself (either explicitly or 
183// implicitly by inspecting the compression method used). I bring this up because, for example, the PVR3 filetype 
184// 'channel type' field does contain size information, but it doesn't need to (and probably shouldn't). 
185// 
186// Example 1. PixelFormat: G3B5R5G3 ChanelType: UNORM 
187// We know the R and B are stored as 5-bit unsigned ints and the G with six bits. We know this from the PixelFormat 
188// alone because it does not contain a 's', 'f', or 'uf'. We further know the intent is to 'normalize' it after 
189// decoding. R would be in [0, 31] and converted to [0.0, 1.0]. The 'U' part of 'UNORM' is redundant because the 
190// pixel-format already told us it was an unsigned integer. 
191// 
192// Example 2. PixelFormat: R11G11B10uf ChanelType: UFLOAT 
193// RG stored as 11-bit unsigned floats (5 exponent, 6 mantissa, no sign bit). B stored as a 10-bit (5,5) float. In this 
194// case the ChannelType is completely redundant because we already know we're using unsigned floats from the 'uf'. 
195// 
196// Example 3. PixelFormat: R8G8 ChanelType: UINT 
197// RG stored as 8-bit unsigned ints (from pixel-format). In this case the ChannelType indicates not to normalize so 
198// each component should be read as an unsigned integer in [0, 255]. 
199enum class tChannelType 
200{ 
201 Unspecified = -1, 
202 UnsignedIntNormalized, 
203 SignedIntNormalized, 
204 UnsignedInt, 
205 SignedInt, 
206 UnsignedFloat, 
207 SignedFloat, 
208 NumTypes, 
209 
210 // Shorter synonyms. 
211 Invalid = Unspecified, 
212 NONE = Unspecified, 
213 UNORM = UnsignedIntNormalized, 
214 SNORM = SignedIntNormalized, 
215 UINT = UnsignedInt, 
216 SINT = SignedInt, 
217 UFLOAT = UnsignedFloat, 
218 SFLOAT = SignedFloat 
219};

255 // Some colour-space component conversion functions. Gamma-space is probably more ubiquitous than the more accurate 
256 // sRGB space. Unless speed is an issue, probably best to stay away from the Square functions (gamma = 2.0). 
257 // 
258 // Colours in textures in files may be in 'Gamma-space' and ought to be converted to linear space before lighting 
259 // calculations are made. They should then be converted back to Gamma space before being displayed. SquareToLinear 
260 // and LinearToSquare are identical to GammaToLinear and LinearToGamme with a gamma value of 2.0. They're a bit 
261 // faster because they don't use the tPow function, only square and square-root. 
262 // 
263 // SquareToLinear will darken the image. Gamma = 2.0 (decoding). Gamma expansion. 
264 // LinearToSquare Will lighten the image. Gamma = 0.5 (encoding). Gamma compression. 
265 float tSquareToLinear (float squareComponent); 
266 float tLinearToSquare (float linearComponent); 
267 
268 // These two are more general versions of the above two functions and use the power function instead of squaring or 
269 // square-rooting. They support an arbitrary gamma value (default to 2.2). For LinearToGamma you are actually 
270 // supplying the inverse of the gamma when you supply the ~2.2 gamma. It takes the invGamma and inverts it to get 
271 // the actual gamma to use. 
272 // 
273 // GammaToLinear will darken the image. Gamma = 2.2 (default/decoding). Gamma expansion. 
274 // LinearToGamma Will lighten the image. Gamma = 1/2,2 (default/encoding). Gamma compression. 
275 float tGammaToLinear (float gammaComponent, float gamma = DefaultGamma); 
276 float tLinearToGamma (float linearComponent, float gamma = DefaultGamma);
